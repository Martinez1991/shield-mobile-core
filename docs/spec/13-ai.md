# 13 — IA Aplicada (ai-svc)

O `ai-svc` transforma "aplicar tudo cegamente" em **proteção adaptativa baseada em risco** — diferencial central. **Não é um LLM voltado ao usuário**; é um pipeline de análise de código.

## Modelos utilizados

| Tarefa | Técnica | Sinais |
|--------|---------|--------|
| Embeddings de código | **GNN** sobre CFG + n-grams de opcode | Estrutura de função |
| Classificação de risco/reversibilidade | **Gradient boosting** | Tamanho, ofuscação prévia, valor |
| Detecção de segredos/PII em strings/símbolos | **LLM pequeno fine-tuned** (self-host) + regex + entropia | `sk_live_`, JWT, chaves |
| Algoritmos cripto | *pattern matching* + classificador | S-boxes AES, constantes SHA, curvas EC |

## Serving
- **ONNX Runtime/Triton**, gRPC, *batch* por build. Modelos **versionados e assinados** (troca gera trilha de auditoria). Sem GPU obrigatória no MVP (modelos pequenos).

## Saída — Risk Map
Para cada símbolo: `risk_score ∈ [0,1]` + recomendações `{virtualize, string-encrypt, rasp-guard}`. O Planner concentra a virtualização (cara) no *top-k* dentro do *budget* de tamanho/latência.

## RAG / Vector DB / MCP / Agentes
- **Não usados no core** (é análise estática, não geração conversacional). *Vector DB* opcional (pgvector/Qdrant) só para busca de similaridade de código entre builds (dedup/cache). **MCP/Agentes/Function calling: N/A** no MVP.
- Se um *copiloto* de policy (LLM assistivo no dashboard) for adicionado no futuro, aplicar a seção "Riscos de IA generativa" abaixo.

## Riscos de IA — OWASP LLM Top 10 / MITRE ATLAS

| Risco | Aplicável? | Mitigação |
|-------|-----------|-----------|
| Prompt Injection | Só se copiloto LLM | Guardrails, separar dados de instruções, allowlist de tools |
| Data Poisoning (feedback loop) | **Sim** | Telemetria assinada/validada; detecção de outliers antes de re-treinar; *human-in-the-loop* na promoção de modelo |
| Model Poisoning | **Sim** | Modelos assinados; proveniência; verificação na carga |
| Sensitive Info Disclosure | **Sim** | **Nenhum segredo detectado é persistido em claro** — só hash/localização |
| Insecure Output Handling | Baixo | Saída do risk map é numérica/estruturada, validada por schema |
| Hallucination | Baixo (não gerativo) | Risk map é sugestão; Planner aplica regras determinísticas |
| Model DoS | Médio | Batch + limites; timeout por build |
| Overreliance | **Sim** | Risk map **assiste**; policy define pisos mínimos; auditável |

## Governança de dados de treino
- Treino só com metadados/derivados (features), **nunca código-fonte do cliente em claro** (PR-10/PR-12). Feedback loop usa telemetria pseudonimizada. *Data lineage* e versionamento de dataset; consentimento por tenant.

## Human-in-the-loop
- Promoção de novo `risk_model` exige revisão + aprovação (não automática); *canary* de modelo comparando cobertura/overhead antes de 100%.
